# 1.4-spec — Vault master password out of page DOM

**Status:** Design / spec. Not yet implemented.
**Replaces / extends:** 1.4 (deferred) + 1.4′ (in-page hardening, explicitly defense-in-depth only).
**Depends on:** 1.1 (SW-only CryptoKey), 1.2 (KDF), 1.13 (run sentinel), F1 (per-step frame probe).

## 1. Threat model — amendment

**Added assumption:** a page the flow runs against may actively attempt
to capture the user's vault master password.

Concrete attacker capabilities we now treat as in-scope:

- Register capture-phase `keydown` / `keypress` / `input` / `beforeinput`
  listeners on `window` / `document` BEFORE any of our code runs.
- Monkey-patch `Element.prototype.attachShadow`, `addEventListener`,
  `HTMLInputElement.prototype.value`, clipboard APIs, etc.
- Poll `document.activeElement` and read `.value` via reflection.
- Inject its own overlay that mimics our modal.

**Consequence:** any unlock surface that renders inside the page
document is compromised by construction. 1.4′ hardened the in-page
modal (capture-phase shield, `isTrusted` checks, `input.value` wipe on
close) and its header comment is explicit that a pre-registered
`window`-capture page listener still wins — that was documented as a
browser event-model limit, not a fixable bug. 1.4-spec is the
structural fix: the master password never enters page DOM.

## 2. Goal

The master password must only ever be typed into a surface whose
document lives in a `chrome-extension://<id>/` origin. The SW (which
already owns the AES key per 1.1) is the only other realm that sees
the password, for the single JS turn it takes to drive `deriveKey` +
AES-GCM verification.

## 3. Non-goals / invariants preserved

- **1.1** SW-only `unlockedKey` stays exactly as-is. `secretsVault.ts`
  API surface (D in the map) does not change. `unlockSecretsVault` is
  still the single derive-and-verify entrypoint.
- **1.2** KDF parameters, transfer payload format, vault migration
  path — untouched.
- **1.13** Fail-close on SW suspension remains the policy. A run that
  is `paused` on unlock when the SW dies becomes `failed` with
  `sw-suspended-during-run` on next cold-start — same as any other
  mid-flight run.
- **F1** Per-step frame resolution is orthogonal; the unlock detour
  happens BEFORE dispatch into content, so frame routing is
  re-evaluated freshly after resume.
- **1.4′** Stays in code as a transitional hardening layer until the
  in-page path is actually removed in 1.4-spec's implementation
  batch. When we delete `promptFlowVaultUnlockOnPage` from
  `flowRunner.ts` the hardened keyboard shield leaves with it.

## 4. Architecture overview

### 4.1 New extension HTML entrypoint

`entrypoints/unlockWindow/` — a minimal WXT HTML entry that renders a
single dialog (master password input + title + context line + Unlock /
Cancel buttons). Mounted at `chrome-extension://<id>/unlock.html`.

Owned by the SAME React stack as the sidepanel so we don't invent a
new UI toolkit. Reuses `ConfirmDialog` / `PasswordPromptDialog`
primitives from `ui/sidepanel/components/` where sensible; a shared
component extraction is **deferred** — duplicate the minimal markup
first, refactor later.

### 4.2 Window opened by SW

SW opens the unlock surface via `chrome.windows.create({
  url: chrome.runtime.getURL('unlock.html?runId=<runId>'),
  type: 'popup',
  focused: true,
  width: 420,
  height: 320,
})`. No new permission required (`windows` comes bundled with `tabs`,
which is already declared in `wxt.config.ts:25`). The returned
`windowId` is stored in SW memory keyed by `runId`.

The URL query string carries **only** the runId — it is an opaque
correlation token, not a capability. Display-worthy context (flow
name, step title, attempt counter, last error) is fetched via an
RPC after mount (4.4).

Window is reused if the same run hits locked vault twice in rapid
succession (should not normally happen — once unlocked the AES key
stays in SW memory until suspend — but defensive: if
`pendingUnlockRequests.has(runId)` already, focus the existing
window instead of opening a second).

### 4.3 Run state — add `paused`

Extend in three places:

- `shared/messages.ts` `FlowRunState` → add `'paused'`.
- `runSentinel.ts` `RunSentinelState` → add `'paused'`;
  `isValidState` predicate grows the case.
- `entrypoints/background/runner/runSentinel.ts` orphan-cleanup
  bootstrap path — `paused` is **treated the same as `running`** for
  orphan broadcasting (produces a `failed` with
  `sw-suspended-during-run`, **not** an attempt to auto-resume).
  Rationale: resuming across SW instances would mean re-opening the
  unlock window unprompted against user intent; fail-close is the
  same policy 1.13 settled on.

### 4.4 New message contracts

All names prefixed `FLOW_RUN_UNLOCK_*` to keep separate from
`SECRETS_*` (which is the general vault API used also by the sidepanel
vault manager). These three sit on top of the existing `SECRETS_UNLOCK`
underneath.

```ts
// MessageType additions
FLOW_RUN_UNLOCK_CONTEXT: 'FLOW_RUN_UNLOCK_CONTEXT',
FLOW_RUN_UNLOCK_SUBMIT:  'FLOW_RUN_UNLOCK_SUBMIT',
FLOW_RUN_UNLOCK_CANCEL:  'FLOW_RUN_UNLOCK_CANCEL',
```

```ts
// Payloads
export type FlowRunUnlockContextRequest = { runId: string };
export type FlowRunUnlockContextResponse = {
  runId: string;
  flowName: string | null;
  stepTitle: string | null;
  siteKey: string | null;
  attempt: number;            // 1-based; increments on wrong password
  lastErrorMessage?: string;  // e.g. 'Invalid master password'
};

export type FlowRunUnlockSubmitRequest = { runId: string; password: string };
export type FlowRunUnlockSubmitResponse =
  | { ok: true }                              // vault unlocked, run resumes
  | { ok: false; code: 'invalid-password'; attempt: number }
  | { ok: false; code: 'run-not-pending' };   // run already finalized

export type FlowRunUnlockCancelRequest = { runId: string };
// Response: { ok: true } — cancel is idempotent; runner handles the
// resulting reject as `secret-vault-unlock-cancelled`.
```

**Retired:** `FLOW_RUN_VAULT_UNLOCK_PROMPT` MessageType and the
`FlowRunVaultUnlockPromptPayload` / `FlowRunVaultUnlockPromptResult`
types are **removed** from `shared/messages.ts`. No legacy
compatibility needed — these are SW↔content messages, both ends
upgrade together in the same release.

### 4.5 Runner-side coordination (`stepExecution.ts`)

- **Delete** `requestVaultUnlockPromptOnPage` (map section A,
  lines 1488–1538).
- **Rewrite** `promptVaultUnlockAndRetry` (map A, 1411–1486): its
  `while(true) { … requestVaultUnlockPromptOnPage … }` loop becomes a
  single `await awaitVaultUnlock(run, step)` call; the retry-on-wrong-
  password loop moves *inside* the unlock window (user retries there
  until they succeed or cancel).
- **New** method `awaitVaultUnlock(run, step): Promise<void>`:
  1. Transition `run.state = 'paused'`; `refreshRunSentinel(run)`
     (so 1.13 observers see the state); `emitStatus(run, true)`.
  2. Register `pendingUnlockRequests.set(runId, { resolve, reject,
     windowId: null, attempt: 0, lastErrorMessage: undefined,
     stepId, createdAt })`.
  3. Open the unlock window; save `windowId` into the map entry.
  4. Return the pending promise. It will be settled by the
     FLOW_RUN_UNLOCK_SUBMIT handler (on success) or the
     FLOW_RUN_UNLOCK_CANCEL handler / `chrome.windows.onRemoved`
     listener (on cancel).
  5. On resolve: `run.state = 'running'`, `refreshRunSentinel`,
     `emitStatus`; caller retries `buildAtomicPayload`.
  6. On reject (cancel): throw `RunnerError
     ('secret-vault-unlock-cancelled', …, { phase: 'execute',
     recoverable: false })`.
  7. On reject (sw-suspended / window-externally-killed during a
     valid unlock that didn't complete SECRETS_UNLOCK): throw
     `RunnerError('secret-vault-unlock-interrupted', …,
     { phase: 'execute', recoverable: false })`. **Note:** 1.4′'s
     `secret-vault-unlock-interrupted` code was previously tied to
     in-page navigation; we reuse the code but widen its meaning.
     Existing `useFlowRunner.ts` i18n branch still applies.

### 4.6 SW message handlers (`bootstrap.ts`)

```ts
case MessageType.FLOW_RUN_UNLOCK_CONTEXT: {
  void respondPromise(async () => {
    const ctx = flowRunnerManager.getPendingUnlockContext(data.runId);
    if (!ctx) return { ok: false, error: 'run-not-pending' };
    return { ok: true, data: ctx };
  });
  return true;
}
case MessageType.FLOW_RUN_UNLOCK_SUBMIT: {
  void respondPromise(async () => {
    const entry = flowRunnerManager.getPendingUnlockEntry(data.runId);
    if (!entry) return { ok: true, data: { ok: false, code: 'run-not-pending' } };
    try {
      await unlockSecretsVault(data.password);  // SW-owned (1.1)
    } catch (error) {
      const msg = error instanceof Error ? error.message : String(error);
      flowRunnerManager.recordUnlockAttemptFailure(data.runId, msg);
      return { ok: true, data: { ok: false, code: 'invalid-password',
                                 attempt: entry.attempt + 1 } };
    }
    flowRunnerManager.resolvePendingUnlock(data.runId);
    return { ok: true, data: { ok: true } };
  });
  return true;
}
case MessageType.FLOW_RUN_UNLOCK_CANCEL: {
  void respondPromise(async () => {
    flowRunnerManager.rejectPendingUnlock(data.runId, 'cancelled');
    return { ok: true, data: { ok: true } };
  });
  return true;
}
```

On `resolvePendingUnlock`: SW calls `chrome.windows.remove(windowId)`
(best-effort) so the window closes cleanly after success. The client
side also closes itself via `window.close()` on `ok: true` as a
fallback — Chrome lets an extension page close its own window without
a permission.

On `rejectPendingUnlock`: SW does NOT call `windows.remove` here;
`onRemoved` is what triggered this path (user-initiated close). In
the explicit-CANCEL-button path (FLOW_RUN_UNLOCK_CANCEL), the unlock
window closes itself after receiving the `{ ok: true }` ack.

### 4.7 Window-close watchdog

One `chrome.windows.onRemoved` listener registered in bootstrap.
Looks up `pendingUnlockRequests` by `windowId`; if still pending:

- If the vault has been unlocked since the entry was created (someone
  else ran `SECRETS_UNLOCK` — e.g. the sidepanel vault manager — in
  the intervening time) → resolve as success. Rare but harmless: the
  run just got what it needed.
- Else → reject as `cancelled`.

This guards the X-button path and also the "Chrome killed the popup
for GPU-process reasons" edge case.

### 4.8 Sidepanel view (`useFlowRunner.ts`)

- `getRunnerStateLabel` (map G, lines 15–31): add a `paused` branch
  with a new i18n key `sidepanel_flow_runner_state_paused`.
- The existing `FLOW_RUN_STATUS` listener needs NO structural change
  — it just receives the new state and the UI re-renders.
- The Run button is already disabled when `runStatus?.state ===
  'running'`; extend the disabled condition to also include
  `'paused'` so the user doesn't accidentally start a second flow
  while the unlock window is up. (`activeRunByTab.size > 0` on the
  SW side is the hard guard; this is UX polish.)
- No Stop button change: `stop(runId)` still finalizes the run as
  `cancelled`; `finalizeRun` clears the pending unlock entry and
  closes the window via `chrome.windows.remove`.

### 4.9 Error-code & i18n surface

Two existing codes keep their names, meanings widened:

- `secret-vault-unlock-cancelled` — now fires for
  window-X-button close OR explicit Cancel button.
- `secret-vault-unlock-interrupted` — now fires for
  SW-kills-window-mid-unlock (previously was page navigation during
  in-page prompt). The user-facing string ("interrupted by page
  refresh or navigation") needs updating to be neutral:
  something like "The unlock window closed before vault unlock
  completed. Please run the flow again." — update all three locales.

One new code:

- `secret-vault-unlock-prompt-unavailable` — already exists from
  the old code path, unchanged. Fires if we can't call
  `chrome.windows.create` (e.g. user denied the extension's window
  permission, though currently we don't require one).

One new i18n key:

- `sidepanel_flow_runner_state_paused` ("Waiting for vault unlock").

Unlock window itself ships its own i18n set for:
- Title ("Unlock password vault")
- Context line template ("Running \"{flow}\" on {site} · step
  \"{step}\" · attempt {n}")
- Password placeholder, Unlock button, Cancel button, invalid-
  password error ("Invalid master password").

All three locales updated atomically.

### 4.10 Content-script impact

**Removed:**
- `promptFlowVaultUnlockOnPage` (map B, lines 339–398) in
  `entrypoints/content/flowRunner.ts`.
- Dispatch of `MessageType.FLOW_RUN_VAULT_UNLOCK_PROMPT` in
  `entrypoints/content.ts` (the case handler and its invocation).
- The capture-phase keyboard shield that 1.4′ added around the
  in-page modal — it's now dead code.

Content script no longer has any involvement in vault unlock.

## 5. Data-flow walkthroughs

### 5.1 Happy path

1. User clicks Run on a flow that references a secret.
2. Runner starts the step, tries to build atomic payload →
   `secret-vault-locked` thrown from `resolveSecretTokens` in
   `secretsVault.ts` (vault is locked, no AES key in SW memory).
3. `executeAtomicStep`'s catch block calls
   `promptVaultUnlockAndRetry`, which calls `awaitVaultUnlock`.
4. Runner: `run.state = 'paused'`; `refreshRunSentinel`; `emitStatus`.
5. Sidepanel receives `FLOW_RUN_STATUS` with `state: 'paused'`;
   renders "Waiting for vault unlock".
6. Runner opens extension window at `unlock.html?runId=…` focused.
7. Window mounts, sends `FLOW_RUN_UNLOCK_CONTEXT` → SW returns flow
   name / step title / siteKey / attempt: 1.
8. User types master password, clicks Unlock.
9. Window sends `FLOW_RUN_UNLOCK_SUBMIT` with password.
10. SW `unlockSecretsVault(password)` derives key, verifies OK,
    stores unlockedKey in SW memory.
11. SW `resolvePendingUnlock(runId)`. Runner's awaiting promise
    settles; runner sets `run.state = 'running'`, refreshes sentinel,
    emits status.
12. SW calls `chrome.windows.remove(windowId)` to close the window.
13. Runner retries `buildAtomicPayload` — now succeeds with resolved
    secret tokens. Step dispatches to content, runs normally.

### 5.2 Wrong password, then right password

8a. User types wrong password, clicks Unlock.
9a. Window sends FLOW_RUN_UNLOCK_SUBMIT.
10a. `unlockSecretsVault` throws "Invalid master password".
11a. SW `recordUnlockAttemptFailure`; responds `{ ok: false,
     code: 'invalid-password', attempt: 2 }`.
12a. Window displays error, increments its attempt counter, focuses
     the password field, clears its value.
13a. Go back to step 8 with correct password.

Runner sees nothing during these retries — the promise stays
pending. The sentinel `attempt` counter isn't stored (it's a display
hint only, kept in SW `pendingUnlockRequests[runId].attempt`).

### 5.3 User clicks Cancel

8b. User clicks Cancel button.
9b. Window sends `FLOW_RUN_UNLOCK_CANCEL`.
10b. SW `rejectPendingUnlock(runId, 'cancelled')`. Promise rejects
     with a runner-internal cancel marker.
11b. Window closes itself.
12b. Runner's `awaitVaultUnlock` throws
     `RunnerError('secret-vault-unlock-cancelled', …)` which
     propagates up through `promptVaultUnlockAndRetry` →
     `executeAtomicStep` → `executeRun`'s catch → `finalizeRun(run,
     'failed', { code: 'secret-vault-unlock-cancelled', … })`.
13b. Sidepanel renders `failed` with the friendly i18n string
     (already exists in `useFlowRunner.ts:74-78`).

### 5.4 User closes window via X

Same as 5.3 via the `chrome.windows.onRemoved` watchdog path. No
difference to the run outcome — `cancelled` error code.

### 5.5 SW suspends while paused

1. Run in `paused`, sentinel on disk carries `state: 'paused'`.
2. SW suspends → all in-memory state gone including
   `pendingUnlockRequests`, including the windowId.
3. Extension window's context is still open on screen but now
   sends FLOW_RUN_UNLOCK_SUBMIT into a revived SW that has no
   record of the run.
4. Revived SW sees `pendingUnlockRequests` is empty → responds
   `{ ok: false, code: 'run-not-pending' }`.
5. Window renders a terminal error ("This flow was interrupted.
   Please close this window and run it again.") and disables the
   Unlock button.
6. In parallel, 1.13 orphan cleanup fires on cold-start. It sees
   the `paused` sentinel, broadcasts `state: failed` with
   `sw-suspended-during-run`. Sidepanel sees the failure.

If the user had already left the unlock window open and doesn't
reopen the sidepanel — no problem. 1.13's `pending_failures`
queue holds the failure notice until they do.

## 7. Security invariants

- **SW is still the only realm that sees the AES key.** Unchanged
  from 1.1. The unlock window passes the password over
  `runtime.sendMessage` which is chrome-extension origin to
  chrome-extension origin — not observable from page scripts.
- **The master password never enters page DOM.** Content script is
  not involved in unlock at all after this batch.
- **Password is never logged.** `unlockSecretsVault` throws an
  "Invalid master password" string on mismatch — no credential
  content in the error. `recordUnlockAttemptFailure` only stores
  the attempt count and the error message string.
- **Window URL carries runId only.** Opaque token. Even if the URL
  leaked (it doesn't), it couldn't be used to unlock without going
  through the SW's message handlers which require a live run.
- **No CSP relaxation.** The unlock window inherits the extension
  default CSP; no external scripts, no `eval`. React build is
  bundled locally as per sidepanel.

## 8. Known scope limits (follow-up candidates)

- **Window focus can be lost.** If the user tabs to another app the
  unlock window drops focus; there's no cross-platform
  "always-on-top" for extension windows. Acceptable — the SW state
  is still `paused`, user can return and finish.
- **Multiple monitors / OS window managers may spawn the window on
  a non-visible screen.** Out of scope; Chrome's `windows.create`
  decides placement.
- **No idle-timeout.** If the user leaves the unlock window open for
  hours, 1.13 will eventually fail-close when the SW suspends.
  Explicit runner-side timeout could be added later; not in this
  batch.
- **No "remember for this run" across steps.** Already irrelevant:
  vault stays unlocked in SW memory across all subsequent steps of
  the same run (and across other runs) until SW suspend.
- **Sidepanel vault manager's own in-sidepanel unlock field still
  exists.** That surface IS extension-origin, so it's not part of
  the page-DOM threat. Unchanged.

## 9. Verification criteria

Pre-merge:

- `npm run compile` + `npm run i18n:check` pass.

Manual smoke (required — same bar as 1.13 / F1):

1. **Happy path.** Record a flow with a `{{secret.X}}` token. Lock
   vault (sidepanel → vault manager → Lock). Run the flow. Unlock
   window opens; enter correct password; run resumes and succeeds.
2. **Wrong password.** Same setup; enter wrong password 2–3 times
   in the unlock window; confirm attempt counter increments and
   error message appears; enter correct password; run resumes.
3. **Cancel button.** Open unlock window, click Cancel; sidepanel
   shows `failed` with the cancelled i18n message.
4. **X-button.** Open unlock window, close via X; same outcome as
   #3.
5. **SW suspension while paused.** Run flow, get unlock window,
   wait (or force-terminate SW from chrome://extensions). Revived
   SW broadcasts orphan failure; unlock window shows
   "interrupted" terminal state; sidepanel renders failed.
6. **Page-DOM verification.** With DevTools open on the target
   page, grep the page's `document.documentElement.outerHTML` for
   any vault-related selectors / text while the flow is paused on
   unlock — should be none. The unlock UI lives entirely in the
   popup window's own DOM.
7. **No regressions.** Sidepanel vault manager unlock still works.
   Non-secret flows run unaffected. Secret resolution across steps
   after unlock continues working without re-prompting.

## 10. Out of scope

- **Consolidating unlock UI between sidepanel vault manager and
  the new unlock window.** Deferred. Initial implementation
  duplicates; extraction can follow once both UIs are stable.
- **Biometric / WebAuthn unlock.** Orthogonal; not blocked by this
  spec.
- **Auto-lock timer.** Still tied to SW suspension; no separate
  timer introduced here.

## 11. Rollout

- Single-batch landing. No feature flag — the in-page prompt is
  deleted in the same commit range that introduces the window.
- CWS in-place upgrade: existing unlocked vaults are unaffected
  (the data layer did not change). Users running a flow at the
  moment of upgrade will see their current flow fail (SW
  re-registered); expected and acceptable.
